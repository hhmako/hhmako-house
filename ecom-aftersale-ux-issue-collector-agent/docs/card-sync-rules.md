# 飞书卡片同步规则

适用对象：体验问题跟进小助手 / 走查问题监控 bot 的体验问题同步卡片。

## 1. 模板原则

- 优先使用已经在飞书卡片搭建器发布的循环版模板。
- 发送脚本只注入真实变量，不修改模板布局、列宽、颜色、间距、按钮样式。
- 如果模板样式不对，需要在飞书卡片搭建器中修改对应组件样式，不要通过字符串伪造。
- 不使用 mock、fallback、默认示例数据补齐卡片。
- 双周合并卡片使用模板 `AAqW1erW8x1lu`。
- 抖音商城问题反馈吐槽群、抖音商城 App 独立端问题反馈群、人工录入的实时问题使用单条卡片模板 `AAqWSrIOa5ECF`。

## 2. 数据来源

每次发送前必须重新读取问题收集表真实记录，不使用缓存。

必须输出或记录：

- Base ID / table ID / 读取时间。
- 总记录数、新录入记录数、已解决记录数、未跟进记录数。
- 每条记录的 record_id、标题、状态、优先级、负责人、来源链接、附件数量、image_key。

## 3. 新录入问题

双周卡片展示过去两周新录入且通过规则的问题，默认最多 5 条。实时来源每条问题单独发送，不混入双周合并列表。

每条必须包含：

- `title_line`：标题，格式为 `P0 · 页面/对象 + 核心异常`。
- `problem_line`：问题，格式为 `问题：具体异常 + 最关键用户影响`。
- `owner_line`：负责人，格式为 `负责人：姓名1、姓名2`。
- `image_key`：当前记录真实高清原帖或 App 截图 image_key。
- `source_url`：真实原帖链接。
- `detail_url`：当前 record_id 的问题表详情链接。

标题要求：

- 模板标题组件必须是富文本或加粗样式组件。
- 如果模板以富文本解析 `**`，发送脚本可以对 `title_line` 使用 `**标题**`。
- 如果模板不解析 Markdown，发送脚本只传纯文本，标题加粗必须由模板组件样式控制。
- 标题最长 24 个中文字符，超长时重新提炼，不硬截半句话。

图片要求：

- 必须来自当前记录对应附件。
- 必须选择最能证明问题的一张。
- 优先使用高清原图，不使用模糊缩略图、封面图、无关图。
- 图片组件必须开启点击预览，点击后能打开飞书大图预览。
- 每条问题使用独立 image_key，不使用模板默认图。

来源按钮：

- 文案固定为“查看来源”。
- 必须跳转当前记录真实原帖。
- 不跳转问题库首页，不跳转多维表格详情。

## 4. 已解决问题

标题展示为：

`已解决${resolvedCount}个问题`

只统计并展示过去两周内状态变为“已解决”的问题。

列表每条只传纯文本：

`【优先级】页面｜问题标题｜负责人`

禁止传入：

- `**`
- HTML
- Markdown 链接
- 蓝色链接样式

## 5. 未跟进提示

模板中使用独立文本变量：

`unfollowed_tip`

发送脚本传入：

```js
unfollowed_tip =
  unfollowedCount > 0
    ? `有${unfollowedCount}个问题暂无跟进，请尽快解决`
    : "";
```

要求：

- 数量来自真实“当前进度 = 待确认”的记录数。
- 数量为 0 时传空字符串。
- 不放入已解决列表。
- 不重复生成。
- 不添加 `**`、Markdown、HTML 或颜色标签。
- 文本颜色由模板组件控制为默认黑色或深色。

不展示“已收录待跟进”“已转业务跟进”“跟进中”的明细列表；这些状态只在表格中保留。

## 6. 底部按钮

底部按钮文案：

`打开问题收集表`

变量：

`issue_base_url`

必须打开真实问题收集表完整 URL。

## 7. 发送前闸门

每次发送前必须通过：

1. `validateRealData()`
2. `validateIssueRules()`
3. `validateRecordMapping()`
4. `validateImages()`
5. `validateLinks()`
6. `validateCardContent()`
7. `validateFinalPayload()`

任一失败都禁止发送。

强制断言：

- payload 不包含旧模板 ID。
- payload 不包含模板默认示例标题、默认负责人、默认图片。
- 新录入/已解决/未跟进数量与真实查询结果一致。
- 每条卡片内容能对应真实 record_id。
- 图片和来源链接来自同一条记录。
- `source_url` 与 `detail_url` 不写反。
- 少于展示上限时，不显示模板默认示例内容。

## 8. 审计文件

发送前生成：

- `raw-records.json`
- `selected-records.json`
- `final-template-payload.json`

这些文件可能含真实数据，必须写入 `tmp/`，不得提交。

## 9. 发送频率和目标群

- 每双周周二 11:00 开始全网搜索并录入体验设计问题。
- 每双周周二 15:30 在“走进一个群就是一家人-交易履约设计小组”群发送双周合并卡片。
- 本轮跑完后，下次只从本轮结束后的原帖日期继续采集，避免重复录入历史内容。
- 抖音商城问题反馈吐槽群、抖音商城 App 独立端问题反馈群、人工填入的问题实时录入，并实时发送单条卡片到同一目标群。
- 卡片发送机器人使用“体验问题跟进小助手”，对应 webhook 从环境变量或本地配置读取，不写入共享包。
