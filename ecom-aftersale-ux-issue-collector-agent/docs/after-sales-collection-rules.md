# 交易售后体验设计问题收录规则

适用对象：交易售后体验问题收集 Agent。

目标：只收录抖音电商或抖音商城 App 产品体验设计相关问题，排除客服态度、商家经营、物流履约、商品质量、纯技术故障和纯情绪吐槽。

## 1. 体验设计问题定义

只有当问题主要由产品自身的界面、交互、信息架构、操作流程、状态反馈、文案表达、内容展示、UI 适配或异常兜底设计不合理导致，并且可以通过修改产品设计方案得到改善时，才归类为体验设计问题。

## 2. 必须收录的设计问题

- UI 布局与适配：遮挡、裁切、重叠、错位、小屏/大字体/安全区适配异常。
- 可读性：文字过小、对比度不足、状态/金额/物流/退款信息被截断。
- 交互：按钮无法点击、点击无反馈、入口难找、输入框被遮挡、页面无法滚动。
- 流程：订单、支付、退款、售后、申诉等流程中断，缺少下一步、撤销、申诉或平台介入入口。
- 信息展示与状态反馈：订单状态、物流签收、退款进度、售后状态表达不清或前后不一致。
- 文案与规则理解：规则说明难懂，按钮名称与实际结果不一致，提示语误导。
- 异常兜底：加载失败、网络异常、接口失败后没有提示、重试入口、进度保留或替代路径。

## 3. 不录入范围

以下问题不作为体验设计问题入库：

- 客服态度、客服回复慢、客服推诿、客服辱骂。
- 商家不发货、虚假宣传、拒绝退款、态度差、售假、承诺未兑现。
- 快递延误、代签、丢件、破损、送错地址、配送员服务态度。
- 商品损坏、缺件、尺寸/颜色/配置不符、商品质量差。
- 纯业务规则争议、赔付争议、政策争议。
- 纯底层技术故障且没有界面反馈或体验兜底问题。
- 无具体页面、流程、截图、录屏或可验证操作描述的情绪吐槽。

一条反馈同时包含外部服务问题和 App 体验问题时，必须拆分，只录入 App 界面、交互、流程、文案、状态反馈和 UI 适配相关部分。

## 4. 截图与录屏主动巡检

只要反馈中包含抖音商城 App 截图、录屏或页面照片，即使用户正文未明确说“遮挡”“错位”“看不清”，也必须主动检查：

- 按钮遮挡核心文字。
- 弹窗内容裁切。
- 文字截断。
- 元素重叠或错位。
- 页面边缘内容被裁切。
- 关键按钮不可见或不可点击。
- 页面无法滚动到底部。
- 大字体、小屏、安全区、刘海屏、折叠屏适配异常。
- 图片、凭证、订单信息显示不完整。

UI 适配问题必须有可观察证据。证据不足时，只能标记为“疑似体验设计问题，需进一步复现确认”，不得作为确定问题推送。

## 5. P0 特殊规则

以下核心任务容器出现裁切、遮挡或关键操作不可见时，直接判定为 P0：

- 退款弹窗、支付弹窗、下单弹窗、售后弹窗、申诉弹窗、身份验证弹窗。
- 弹窗高度超过屏幕，底部内容被裁切。
- 用户无法看到完整选项。
- 提交、确认、下一步等关键按钮不可见。
- 关键按钮位于屏幕外，且没有明确证据证明可滚动到达。
- 弹窗内部无法滚动，或没有明确滚动提示。
- 系统安全区、手势条或导航栏遮挡操作区域。
- 用户无法完成退款、支付、售后或申诉等核心任务。

不得仅因为“页面仍能关闭”或“App 其他功能仍能使用”而降级为 P1。判断核心是当前关键任务是否能顺利完成。

- 订单列表 / 订单详情：余卉、马慧洋。
- 物流详情：龚戬。
- 售后申请 / 售后详情 / 退款进度 / 优惠券 / 运费险 / 权益：杨依依、程瑶。
- 商家客服 / 平台客服：孙霄、聂广艺。
- 国补 / 珠宝鉴定 / 以旧换新：石静雯。
- 安装服务 / 维修服务 / 时效：黄樱。
- 无法判断：待分配。

补充规则：

- 退款和退款寄件属于售后问题，不归入物流履约。
- 客服物流卡片文字截断属于物流详情/物流卡片显示问题，负责人按物流归属。
- 除售后/权益之外的问题，不展示或使用“发送到售后专家”相关字段和按钮。
- 下单页面不属于交易售后收录范围，除非问题发生在订单/售后/物流/客服链路中。

## 7. 问题分类四分类

当前“问题分类”字段只能保留以下 4 个选项，不新增重复字段：

1. 界面显示与适配问题
2. 信息表达与理解问题
3. 操作交互与反馈问题
4. 任务流程与闭环问题

判断口诀：

- 看不全、显示错 -> 界面显示与适配问题
- 看不懂、易误解 -> 信息表达与理解问题
- 不好点、没反馈 -> 操作交互与反馈问题
- 做不完、没后续 -> 任务流程与闭环问题

判断顺序：

1. 直接问题是遮挡、裁切、重叠、错位、屏幕适配，归为“界面显示与适配问题”。
2. 直接问题是点击、输入、入口或操作反馈，归为“操作交互与反馈问题”。
3. 核心是无法完成完整任务、流程中断或缺少后续路径，归为“任务流程与闭环问题”。
4. 核心是文案、规则、状态或信息难以理解，归为“信息表达与理解问题”。

必须优先根据直接设计原因分类，不只按最终影响分类。

## 8. 当前进度

当前进度字段只允许以下 6 个选项：

- 待确认
- 已收录待跟进
- 已转业务跟进
- 跟进中
- 已解决
- 不采纳

新录入记录默认写入“待确认”。

卡片同步规则：

- 两周内新录入且符合收录规则的问题，进入“双周新录入”卡片。
- 两周内状态为“已解决”的问题，进入“双周已解决”列表。
- 当前状态为“待确认”且无人跟进的问题，只在卡片底部展示数量提醒。
- 当前状态被设计师人工调整为“不采纳”的问题，不再更新到同步卡片。
- “已收录待跟进”“已转业务跟进”“跟进中”用于表内过程管理，不作为双周卡片的独立列表展示。

## 9. 来源与截图

- 来源必须写清楚具体平台或具体群名。
- 机器人自动采集可写来源平台；人工补录应标记为“人工录入”。
- “查看来源”必须跳转原帖：小红书原始笔记、微博原始帖子、哔哩哔哩原始视频/动态、吐槽群原始消息定位。
- 新录入问题必须带原帖截图或原始 App 截图，图片需要能直接证明问题。
- 对吐槽群消息，如果定位链接不稳定，需要把吐槽群原消息截图放入原帖截图字段。
- 小红书、微博、哔哩哔哩需要尽量保持登录态可用；若小红书命中风控，应使用合规替代检索方式继续寻找体验设计问题，不得绕过平台授权或伪造来源。
- 全网问题录入时按原帖日期从远到近录入；新录入记录必须写入原帖日期，并追加到对应场景子表。
- 新录入问题必须写入对应子表；“全部视图”只作为全部记录汇总。全部视图下可使用“场景”多选标签，只有百分百确认命中多个场景时才多选。

## 10. 录入前后校验

录入前必须判断一次是否为体验设计问题；录入后再检查一次。

不通过以下任一项时不得正式入库：

- 是否发生在抖音商城 App 页面或产品流程中。
- 是否由界面、交互、流程、文案、状态反馈或适配导致。
- 是否可以通过修改 App 产品设计得到改善。
- 是否不是客服、商家、商品或物流履约问题。
- 是否存在文字、截图、录屏或可验证操作证据。

证据不足时标记为“待确认”或“疑似体验问题，需进一步复现确认”。

补充排除规则：

- 生活服务、本地生活等非电商交易售后问题不录入；如误录，应删除并记录原因。
- 端技术问题本身不录入；只有技术异常后的提示、重试、兜底路径等体验设计问题可单独拆分录入。
- 无有效截图或截图无法证明问题的记录不录入。
- 商家未填写退回原因、商家处理态度、商家履约动作导致的问题不录入；只有 App 页面没有说明、入口或兜底路径时才拆分录入体验设计问题。
